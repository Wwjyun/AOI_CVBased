# AOI CV Based 週報

日期：2026-07-16（星期四）至 2026-07-22（星期三）

本週完成 AOI 共用 CPU／CUDA 執行架構、產線驗證工具鏈、GUI 與 CPU 工作流程優化、Detector 401 效能分析、401-2 既有判定契約恢復、CSV 面積單位換算，以及 NG Tile 面積分類工具與兩項 Windows 發布。期間共完成 62 筆提交，淨異動 90 個檔案，新增 12,771 行、刪除 910 行；週期末版本為 VisionFlow AOI v1.1.1，完整測試套件增至 151 項。

## 本週完成事項

### 1. CPU／CUDA 共用架構與執行正確性

- 建立 immutable `PreprocessPlan` cache、tensor contract 與 backend capability report，使 CPU、CUDA 與 fallback 路徑可被一致追蹤與驗證。
- 將 Detector 401、401-1 與 900 遷移到 shared preprocess plan；Detector 900 支援 CPU DAG／multi-output，避免重複 Gray 前處理。
- 完成 generic persistent CUDA linear plan、DAG multi-output ABI、resident image／device ROI 與 VRAM-aware ROI batch 原始碼與 Python OOP routing，降低重複傳輸、配置與同步成本。
- 統一 `auto`、`cpu`、`cuda` 模式語意：CPU 模式不載入 CUDA、auto 可安全 fallback、strict CUDA 失敗時不得靜默改走 CPU。
- 修正 generic native plan 遇到 `morph_operation: none`、零 iterations 或 1×1 kernel 時的錯誤判定，不需重編 CUDA DLL 即可維持 CPU／primitive 語意。

### 2. 驗證、可觀測性與產線安全

- 新增 CUDA event phase timings，分離 H2D、D2D、kernel、D2H、Gaussian、Adaptive Mean、Threshold 與 Morphology 等耗時。
- 建立 cold／warm、median／P95、VRAM、allocation、Driver／Toolkit／GPU 與 commit 資訊的可重現 benchmark schema。
- 補齊 context reuse、OOM／error recovery、不同 shape／channel／參數、64²～1024² crossover、Morphology iterations 與 10／100／1000 次壓測工具。
- 建立五份 production recipes 的 acceptance manifest、Windows CPU CI 與 RTX 3090 self-hosted workflow；加入 Nsight 可選擷取、48 小時 heartbeat 與 P95 regression gate。
- 完成 strict detector schema、recipe／build SHA-256、Git commit provenance、NG dataset sidecar、合成 golden cases、Python 3.13 Windows dependency lock 與 Hypothesis preprocess fuzz。

### 3. GUI、CPU 吞吐與操作流程

- 完成 P9 CPU／輸出優化：recipe cache、opt-in tile 平行、batch thread budget、週期 GC、NG 圖片平行寫檔、overlay 格式／品質／縮圖、debug preprocess images 與結果 TypedDict 契約。
- 改善 GUI 全域／步驟／事件狀態層級，顯示實際 backend 與 fallback notice；補強 NG 鍵盤導覽、Recipe dirty／validation、QSettings 工作環境回復，以及 Batch／Monitor model-view 與 deterministic scatter sampling。
- GUI 單張執行可依 Recipe 重用 `GpuExecutionSession`，並分列 detector、pipeline-before-report、reporting、end-to-end 與 user-wait 時間。
- 新增 OP／工程／管理模式密碼授權；程式啟動固定從 OP 模式開始，避免自動恢復高權限狀態。

### 4. Detector 401／401-2 效能與相容性

- 新增 Detector 401 全圖 profiling harness，分離 template matching、ROI generation、detector、reporting、CUDA events、kernel launches 與 peak working set。
- 新增離線 profile analyzer，可檢查 PASS／NG、座標與 fallback gate，並依 launch／ROI、同步、Morphology、D2H、Gaussian、Adaptive Mean、CPU contours 等證據排序優化方向。
- 依既有操作契約，最終將 401-2 恢復為「經面積篩選後、逐 contour 計算白像素比例」；保留單一 Tile 多筆 defect、bbox、area、排序與 metadata。
- 7/21 已在具 NVIDIA GPU 的機器確認 CPU 與 GPU 模式皆可執行，但兩者整體耗時相近；目前僅能證明可執行，尚不能宣稱具加速效益。

### 5. 面積校正與 NG Tile 分類工具

- Recipe Designer 新增「精度（µm/px）」設定，CSV 依 `area_px × pixel_size²` 集中換算為 `um²` 並輸出 `area_unit`；舊 Recipe 未設定時維持 `px²`，不改變 Detector 判定、JSON 或 GUI 結果。
- 新增獨立 `export_ng_tiles_by_area.py` GUI／CLI 工具，可依 CSV 面積區間分類並複製 NG Tile，支援同 Tile 最大／總和／最小面積、未分類資料夾、JSON sidecar 與舊 CSV 相容。
- 面積分類工具採獨立版本與輸出目錄，不移動 AOI 原始結果，並避免 px²／µm² 混合分類。

### 6. 文件與版本發布

- 新增功能驗證與版本控制報告，整理架構、Detector、GUI、輸出追溯、CPU／GPU／fallback、測試、CI、限制與後續規劃。
- 完成並發布 NG Tile 面積分類工具 v1.0.0 one-file Windows 包、繁中說明與 packaged smoke；現有 9 張 NG Tile／9 份 JSON 實際分類通過。
- 發布 VisionFlow AOI v1.1.1 Windows x64；本版明確採 CPU-compatible 配置，缺少 CUDA DLL 時可安全 fallback。
- 建立 `ng-tile-area-tool-v1.0.0` 與 `v1.1.1` 兩個版本 Tag 與對應 GitHub Release。

## 目前限制與風險

- 尚缺固定且可追溯的 production 測試集；五份 Recipe 各一張 PASS／NG，共 10 個案例仍待提供，因此 CPU／GPU 全流程等價與量產驗收尚未定案。
- GPU 實機觀察尚未固定影像、Recipe、cold／warm 次數與環境資訊，也未取得 median／P95、完整 profiler JSON 或至少 1.5 倍加速證據。
- 目前開發機缺少 `nvcc`、MSVC C++ toolchain 與 NVIDIA CUDA 裝置，無法針對 v1.1.1 commit 重建並完成 DLL／ABI／錯誤注入／長時間壓測。
- 401-2 已由自動測試固定既有逐 contour 語意，但仍需真實 production PASS／NG 案例驗收。
- Weekly Windows Packaging 的 packaged smoke 尚未轉綠；在排程 CI 重跑成功前，手動發行驗證與自動發行驗證需分開看待。
- VisionFlow AOI v1.1.1 為 CPU-compatible 發行；NG Tile 面積分類工具 EXE 尚未簽章，對外部署時需標示 SmartScreen／信任提示。

## 下週計畫

1. 建立並凍結五份 Recipe 共 10 個 PASS／NG production cases，補齊 expected final、defect、bbox、area 與 metadata。
2. 在 RTX 3090 記錄 Driver、VRAM、CUDA Toolkit 與編譯器版本，以 `sm_86` 重建 DLL／LIB／測試 EXE，並檢查 exports、dependencies 與 native smoke。
3. 以相同 image／Recipe／輸出條件重跑 Detector 401 cold 1 次、warm 10 次與 GUI 10 次，保存 profiler／analyzer JSON、median／P95 與 fallback 證據。
4. 只依 profiling 結果選擇下一步：launch／ROI gather／D2H 為主則評估 ROI batch；Morphology 為主則評估 separable／shared-memory；證據不足時維持 production CPU 預設。
5. 完成 GPU error／OOM、8／16／32／64 ROI batch、100／1000 張 VRAM 穩定性，以及有 GPU／無 GPU 的打包驗證矩陣。
6. 修復並重跑 Weekly Windows Packaging 的 packaged smoke；使用真實 AOI 輸出驗收 µm² 換算與 NG Tile 分類，並補齊 CPU-compatible、CUDA DLL 與未簽章工具的發布說明。
